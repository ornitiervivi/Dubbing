# Design Patterns Guide
Patterns são ferramentas, não decoração. Recomendados quando reduzirem complexidade: Strategy (providers IA/storage/tradução/transcrição/TTS/render), Factory (objetos complexos com invariantes), Builder somente se reduzir complexidade e sem record, Adapter para providers externos, Repository por boundary de persistência, Gateway/Port para integrações, Specification com ganho real, Template Method para algoritmo estável, Chain of Responsibility para pipeline/validação com menor acoplamento, State/política explícita para DubbingJobStatus, Command para entrada de use case, Mapper entre camadas sem regra pesada, Facade para subsistemas externos, Domain Service apenas quando regra não cabe em entidade/value object.
Perigos/proibidos: AbstractFactory sem necessidade, Manager/Helper/Util genérico, service gigante, domínio anêmico sem necessidade, overengineering, pattern por estética, strategy com implementação única sem perspectiva real, interface para tudo, factory trivial, mapper com regra de domínio, use case que faz tudo.
